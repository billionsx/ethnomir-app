### 3.6. Модуль М05. Чеки и QR-экосистема

Сквозной модуль квитирования всех платных операций приложения. Реализован двумя компонентами: `OrderView` (полноэкранный просмотр чека по публичной ссылке, строки 10151–10310) и view-режим `receipts` внутри `PassportView` (личный список пользователя).

- **Каждая покупка** (билет, бронь, заказ, мастер-класс, сертификат, доставка) сохраняется одновременно в двух таблицах БД: `orders` (с полем `order_code`) и/или `receipts` (с полем `receipt_code`). Связанные позиции хранятся в `receipt_items`.
- **Уникальный код чека** генерируется при создании заказа и используется в публичной ссылке. Формат кода — короткий человекочитаемый идентификатор (например, `EM-XXXXX`); конкретный шаблон управляется через триггеры БД и может быть скорректирован Заказчиком в рабочем порядке.
- **QR-код** размером 200 × 200 пикселей для бесконтактного считывания на кассе / в отеле / в ресторане. Кодирует публичную ссылку.
- **Публичная ссылка** в формате `ethnomir.app/#order/<code>` — открывается без авторизации и может быть передана третьему лицу (бухгалтерии, родителям, коллегам). Доступ обеспечивается RLS-политикой `orders_public_select_by_code` и `receipts_public_select_by_code`.
- **Содержимое экрана чека:** номер, статус (`pending` → «Ожидает оплаты», `confirmed` → «Подтверждён», `paid` → «Оплачен», `completed` → «Завершён», `cancelled` → «Отменён», `refunded` → «Возврат»), дата создания, итоговая сумма, состав заказа (позиции с количеством и ценой), способ оплаты (заявка / наличные / карта), реквизиты парка (юр. наименование, ИНН/ОГРН/КПП, адрес, телефон, email — из таблицы `park_info`), QR-код для кассира.
- **Дизайн** — авиационный boarding pass в Liquid Glass-стилистике: тёмная шапка с брендом «Этномир», крупный номер, QR в рамке, метрики чека.

### 3.12. Модуль М11. Отзывы и NPS

- **Источник данных** — таблица `reviews` со связью на сущности (`item_type`, `item_id`); специализированная таблица `restaurant_reviews` для ресторанов; для отелей — `hotel_reviews`. Модерация — поле `moderation_status` (`pending` / `approved` / `rejected`).

- **NPS (Net Promoter Score — индекс лояльности клиентов)** — автоматический подсчёт по формуле NPS на основе ответов гостей после посещения; цель — 91 % и выше.
- **Интерфейс добавления отзыва** — рейтинг (1–5 звёзд), текст, прикрепление фото (опц.); начисление + 50 баллов за отзыв с фото.
- **Ответ администрации** — каждый отзыв может получить официальный ответ через CRM-модуль «Reviews» (поля `manager_response` и `response_at`); ответ виден всем гостям под отзывом.
- **Модерация** — автоматическая фильтрация ненормативной лексики плюс ручная модерация в CRM.

### 3.13. Модуль М12. Уведомления

Три канала доставки и сквозной центр уведомлений:

- **Push-уведомления** через Firebase Cloud Messaging: таблицы `push_subscriptions`, `push_tokens`, `push_messages`. Шаблоны и расписание — в CRM-вкладке «Notifications».
- **SMS** через сервис SMS-отправки (SMS.ru или аналог): таблица `sms_rate_limits` обеспечивает rate-limiting (защита от спама и злоупотребления OTP); SMS используется для phone-OTP при входе и для критичных транзакционных сообщений.
- **In-app центр уведомлений** — оверлей, открывается из шапки Паспорта по иконке колокольчика. Источник — таблица `notifications` (для клиентского контура) и `crm_notifications` (для сотрудников).

**Типы уведомлений:**

- маркетинговые (акции, скидки, бонусы) — таблицы `promos`, `promo_banners`, `promo_codes`, `promo_code_uses`;
- транзакционные (подтверждение оплаты, напоминание о брони, изменение статуса);
- операционные (изменение времени мастер-класса, отмена события, готовность заказа).

- **Поиск (`crmGlobalSearch`)** — единая строка над панелью; через RPC `crm_global_search` ищет по всем сущностям (брони, заказы, отзывы, гости, лиды, штат, задачи); тап на результате переключает на нужный модуль и подсвечивает запись.

| Линейный персонал | `receptionist`, `cashier`, `concierge`, `housekeeper`, `waiter`, `kitchen`, `delivery`, `support`, `content_manager`, `security`, `staff` |

**4.7. Иерархия доступа** (минимум 5 ступеней по убыванию объёма прав):

1. **Владелец** (`owner`) — видит и меняет всё;
2. **Директор** (`director`) — все модули, кроме конфиденциальных финансовых деталей по сотрудникам;
3. **Руководитель подразделения** (`*_head`) — модули своего домена + просмотр смежных;
4. **Линейный сотрудник** — операции своего участка (кассир видит кассу и заказы, консьерж видит брони и заявки, и т. п.);
5. **Партнёр** — только свой объект (один отель, один ресторан и т. п.).

**7.5. ## 9. Блок 1. Мега-приложение. Безопасность, приватность, compliance

**9.1. Соответствие 152-ФЗ «О персональных данных»:**

- явное согласие пользователя на обработку ПДн при регистрации (с указанием версии документа, даты, IP, устройства — хранится в таблице `consents`);
- хранение ПДн на серверах, соответствующих требованиям 152-ФЗ (на момент запуска — в соответствии с актуальными практиками; при требовании — перенос на РФ-серверы по отдельному соглашению);
- возможность отзыва согласия через настройки Паспорта с автоматической анонимизацией данных в связанных таблицах;
- журнал действий с ПДн — в отдельной таблице аудита.

**9.2. Row Level Security (RLS)** — политики безопасности на уровне СУБД, проверяющие каждое обращение к данным в зависимости от роли пользователя. Распределение политик по типам операций (чтение, запись, обновление, удаление, общие политики на все операции) реализуется в объёме, достаточном для обеспечения перечисленных выше ролей и ограничения доступа в соответствии с этими ролями.

**9.3. HMAC-подпись чеков** — каждый чек и QR-код криптографически подписан; ключ хранится в `Vercel encrypted env`.

**9.4. OTP-авторизация** — вход по одноразовому коду (SMS или e-mail), без паролей. Rate-limiting для защиты от перебора.

**9.5. Шифрование** — весь трафик по HTTPS с TLS 1.2+; секреты хранятся в защищённом хранилище переменных окружения облачной платформы и не попадают в репозиторий исходного кода.

**9.6. OWASP-стандарты:**

- защита от SQL-инъекций (параметризованные запросы, ORM);
- защита от XSS (escape пользовательского ввода);
- защита от CSRF (токены);
- защита от brute-force (rate-limiting);
- регулярный аудит зависимостей на известные уязвимости.
