# Договор на разработку ethnomir.app — рабочий черновик

Папка для итерации проекта рабочего договора между Billions X и ООО «ЭТНО-ДЕРЕВНЯ» по разработке и внедрению суперприложения **ethnomir.app**, ре-упаковке сайта ethnomir.ru, ре-брендингу Этномира и продуктовой линейке.

Статус: **черновик v3.0 · апрель 2026**. Дата подписания в теле договора: **07 мая 2026** (Мастер-число 22 по нумерологии — символ материальной реализации масштабных идей, четверг — день Юпитера, традиционно благоприятный для юридических и договорных дел).

## Изменения в v3 (большой пакет правок)

### Договорная часть

1. **Дата договора** — 07 мая 2026.
2. **Ответственность сторон за просрочки** — пени Заказчика и пени Исполнителя полностью удалены. Остался общий принцип «по законодательству РФ», предел ответственности (сумма Договора), исключение косвенных убытков, исключение ответственности за финансовые результаты Заказчика.
3. **Порядок сдачи-приёмки (раздел 5)** — существенно упрощён. Убраны жёсткий 10-дневный срок на приёмку и требование «мотивированного отказа». Теперь: уведомление → рассмотрение в разумный срок → замечания в рабочем порядке по e-mail/мессенджеру → устранение по согласованию сторон. Если стороны не приходят к согласию в течение 30 дней при отсутствии существенных недочётов — этап считается принятым.
4. **Сроки устранения дефектов (раздел 10)** — увеличены примерно в 5 раз, без разделения на рабочее/нерабочее время. Критичный: первичный ответ в течение 24 часов, устранение до 10 календарных дней. Некритичный: первичный ответ в течение 5 дней, устранение до 50 календарных дней. Запрос-доработка: первичный ответ в течение 10 дней, реализация по отдельному соглашению.
5. **Описание этапов (раздел 4)** — упрощено. Убрана разбивка «2 мес. + 2 мес. + 1 мес.» внутри этапа. Теперь: общий объём работ этапа + срок этапа + результат этапа + закрытие.

### Техническое задание (Приложение №1)

6. **AI-чат поддержки (М06) — удалён полностью.** Модули перенумерованы: М07→М06 (Этномир-хаб), М08→М07 (Франшиза), М09→М08 (Календарь), М10→М09 (Сертификаты), М11→М10 (B2B), М12→М11 (Отзывы), М13→М12 (Уведомления), М14→М13 (Поиск). Итого **13 клиентских модулей**.
7. **PRO-подписка — убрана** во всех местах (таблица экранов Паспорта, способы заработать баллы, типы уведомлений, отдельный раздел 5.3). Паспорт теперь содержит 15 экранов вместо 16.
8. **Баллы и правила геймификации** — помечены как примерные. Добавлено явное условие: «Указанные значения — примерные и приведены для иллюстрации логики программы лояльности. Точные правила определяются и корректируются Заказчиком в рабочем порядке через CRM без изменения настоящего Договора.»
9. **QR-экосистема** — убрано число «минимум 1 203 QR-кода». Теперь — перечисление сценариев применения (открытие стран и регионов, мастер-классы, рестораны, отели, вход, партнёры) с оговоркой: «точное количество определяется в рабочем порядке».
10. **Производительность** — поднята: пиковая 50 000 одновременных пользователей (было 5 000), штатная 20 000 (было 1 500).
11. **VoiceOver / TalkBack / WCAG 2.1** — удалены. Осталась общая формулировка про «современные практики проектирования».
12. **Технологический стек (раздел 7)** — полностью переписан как **ориентировочный**. Зафиксировано право Исполнителя адаптировать стек по ходу Работ по мере приближения к требованиям App Store и Google Play без дополнительного соглашения. Конкретные сервисы (Next.js, React, Vercel, Supabase, ЮKassa, СБП, Firebase Cloud Messaging, PostHog) упоминаются **только в качестве примеров** — «например, Vercel или аналогичный», «например, ЮKassa».
13. **Все прямые ссылки на имена таблиц БД** (`crm_leads`, `audit_trail`, `wallet_transactions`, `promos`, `push_subscriptions` и т. д.) — заменены на русскоязычные описания по смыслу. Технические идентификаторы убраны из договорного текста.
14. **«Контент BillionsX»** как отдельный домен базы данных — **убран из раздела 8.3**.
15. **Количественные показатели БД (раздел 8.2)** — теперь «ориентировочные» вместо «минимально обязательных». Добавлена оговорка, что конкретные значения на момент сдачи могут отличаться в любую сторону в зависимости от эволюции архитектуры.
16. **RLS-политики (раздел 9.2)** — убраны жёсткие числовые минимумы (было «230 минимум», разбитые по типам операций). Теперь — формулировка по смыслу: «в объёме, достаточном для обеспечения перечисленных ролей».
17. **GitHub и конкретные репозитории** — убраны из тела договора.
18. **Supabase project ID** — убран.
19. **Мониторинг в критериях приёмки (раздел 14.1)** — смягчён: вместо «monitoring-score ≥ 95 %, 20/20 деплоев READY» теперь «инфраструктурный мониторинг показывает стабильное состояние системы».
20. **Четыре столпа** (Геймификация, Дашбордность, Монетизация, Вдохновение) — добавлены в преамбулу ТЗ (пункт 1.7) в соответствии со справочником продукта.

### Типографика и стиль

21. **Шрифт всего DOCX** — SF Pro Display для заголовков, SF Pro Text для тела текста. Обеспечена принудительная установка rFonts на каждый run (ascii / hAnsi / cs / eastAsia) — кириллица гарантированно берёт SF Pro.
22. **Разделительные линии `---` удалены** — 44 разделителя. Строгий стиль без горизонтальных правил.
23. **Цвет текста — только чёрный.** Нет акцентных цветов, нет заливок ячеек таблиц, нет подсветок. Строгий ч/б.
24. **Проверка DOCX**: 1817 runs, 100 % покрытие шрифтами SF Pro Display / SF Pro Text, 0 цветных акцентов, 0 заливок, 0 горизонтальных правил.
